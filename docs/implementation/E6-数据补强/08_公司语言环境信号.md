# E6-08 公司语言环境信号(帮英语弱的人找语言友好雇主)

> 缘起:2026-07-22 Frank「最好能知道这个公司是不是以华人为主…这样英语不好的人也有优势」「韩国人找韩国公司」。
> 定位:公司详情页(E8-09 B,#182/#184)加一维**语言环境**信号——中文用户找中文友好、韩语用户找韩语友好,卡住「英语不好也有优势」。
> 约束(Frank 同日两条):① **走 ETL 单独抓,别在展示层临场 hack**;② 本件=设计文档,**未获批不动代码**。

## 1. 一条红线(整件的宪法):语言环境,不是族裂判定

- **不做**「这家员工是不是华人/韩国人」的族裔断言——既没有可靠依据、又敏感(可能扯上歧视),更违反本项目「宁可留空也不瞎猜」。
- **只做**可核验的**语言环境**信号:这家的**招聘/服务用什么语言**。抓手是雇主自己写在帖子里的东西(要求某语言、某语言加分),以及公司名的语言。这些是**观察到的事实**,不是对人的推断。
- 证据不足 → **留空**(不显),绝不硬标。标出来的一律带「依据帖子语言线索」口径,不当硬事实。

## 2. 信号定义(值域小、可留空)

单字段 `lang_affinity`(companies 新列),枚举:

| 值 | 含义 | 判定依据(全部来自已抓数据,零 LLM) |
|---|---|---|
| `zh` | 中文友好 | 该司在招帖正文/语言字段命中 Mandarin/Cantonese/普通话/粤语/中文,**或**公司名含 CJK/常见拼音商号(粗筛,见 §3) |
| `ko` | 韩语友好 | 同上,命中 Korean/한국어/한국 |
| `''`(空) | 无信号 | 未命中任何非英语语言线索(**默认**,绝大多数公司) |

- **只做 zh/ko**(对齐本站三语受众;fr 是官方语言不算「优势」信号,en 是默认不标)。
- 一家可同时中文+韩语?罕见;真命中就都标(字段可扩成小数组,v1 先单值,取命中最强的一个)。
- **粗筛非认定**:标了只代表「这家帖子出现过该语言线索」,不代表全员说该语言。措辞灰注讲清。

## 3. 数据来源与判定(ETL 本职,纯文本可批量)

**输入**(全是已落库/已抓的,零新增外呼):
- 职位帖正文 `jobs.description`(05b 已抓)+ 结构化语言字段(若 05b 解析出 languages 节)。
- 公司名 `companies.name`。

**判定逻辑**(白名单正则,镜像既有清洗脚本的克制):
1. **帖内语言线索**(主信号,最硬):该司**任一在招帖**的 description 命中 `\b(mandarin|cantonese)\b`、`普通话|粤语|中文` → `zh`;命中 `\bkorean\b|한국어|한국` → `ko`。理由:雇主主动写「Mandarin an asset / 要求普通话」是最强的语言环境证据。
2. **公司名语言**(弱信号,兜底):名含 CJK 字符,或命中**高置信中/韩商号词表**(如 `Wok|Dumpling|Szechuan|Hakka` / `Kimchi|BBQ Korea`…,人工小词表,宁缺勿滥)→ 对应语言。**单靠名不足以定**,仅当帖内无线索时作弱兜底,且灰注标「据公司名推测」。
3. 两者都没有 → 空。

**步骤落位**:新建 `etl/clean/05e_lang_affinity.py`(clean 层,一个关注点一个脚本)。
- 顶部声明 `IN_*`/`OUT_*` 全路径常量(经 `_paths` 解析,运行时打印),与其他 clean 脚本同规范。
- 读 postings + companies,按 company 聚合其在招帖文本 → 算 `lang_affinity` → 回写公司维度。
- **纯文本、零 LLM、零外呼、可整表批跑**——不碰 lazy-first 铁律(那条是禁「批量预抓**外部/LLM** 公司数据」;本步是本地正则,不受限)。
- 进 `09_build_mart.py` 的 companies 表列 → seed 入库(companies 加 `lang_affinity` 列,DDL 先行)。

**实现前先跑一次校准**(feasibility):统计各语言线索在在招帖里的命中量级,定阈值、看噪声(如 "Chinese" 单词可能指菜系不指语言 → 只认 Mandarin/Cantonese 不认泛 "Chinese")。

## 4. 展示(公司详情页 + 在招职位;克制,不喧宾夺主)

- **公司页头部**:命中时加一枚 chip「中文友好」/「韩语友好」(与「知名企业」chip 同排)。灰注一行:`依据招聘帖语言线索,非族裔判定`。空则不出(寸土寸金)。
- **在招职位板筛选**(下一步,可选):筛选器加「语言友好」项,让 zh/ko 用户一键筛——这才是「找语言友好雇主」的真正闭环。v1 先出公司页 chip,筛选下批。
- i18n 三语键:`co.langZh`=中文友好 / `co.langKo`=韩语友好 / `co.langNote`=依据招聘帖语言线索,非族裔判定。

## 5. 红线落地

- 证据不足留空;chip 措辞是「语言友好」不是「华人公司」;灰注必带「非族裂判定」。
- 词表人工维护、宁缺勿滥(镜像城市译名人工表教训:让模型判「有无」不可靠,有限集人工定表)。
- 零新增外呼、零 LLM、可批量——ETL 本职,不占朋友 qwen,不破 lazy-first。
- closed 岗不参与判定(只看在招帖,信号随雇主当前招聘而动)。
