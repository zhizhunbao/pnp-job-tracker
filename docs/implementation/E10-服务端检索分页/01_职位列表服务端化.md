# E10-01 职位列表服务端化(分页 + 检索 + 筛选下沉,干掉 20k blob)

> 2026-07-17 Frank 看「显示更多(还有 19950 条)」与头条 32507 对不上,连拍三条:
> 「改成后端分页」「一次查 2 万条开玩笑呢」「检索也改成后端 拼 SQL 检索」。
> 拍板决策(本 session 三问):① 分页=OFFSET/LIMIT(用户不深翻,keyset 留后);
> ② 个性化(匹配视图 / FOMO 计数 / 推荐)=**独立端点保全**(SQL 预筛→TS 跑 match(),功能不降级);
> ③ 搜索=**核心列**(职位/公司/省市区/NOC 码/来源 走 ILIKE;分类中文标签后续 query→码映射补)。
> **本文档=动手前的设计,过目签字后再改代码。**

## 1. 背景与病灶

- 现状:SSR 出首屏 50 行,客户端 `/api/jobs-data` **一次性 `LIMIT 20000`** 全捞(≈25MB),
  `JobsTable` 在这个 blob 上做**全部**筛选/搜索/排序/分页(`rows` useMemo)。
- 病灶:① 库里在线岗 **32437**(实测)> 载入上限 20000 → 1.2 万岗翻不到,「还有 19950」与头条 32507 对不上,像 bug;
  ② 每个访客首屏后要吞 25MB,与刚做的 SSR 瘦身背道而驰;③ blob 越大越卡,不可持续。
- 根因:**为了「筛选瞬时」把全量塞前端**。正解=筛选/排序/分页全下沉 SQL,前端只拿当前页。

## 2. 目标与非目标

- 目标:职位列表的 **筛选 + 搜索 + 排序 + 分页 全部服务端(SQL)化**;前端每次只持有「当前累计页」(50×N),
  `/api/jobs-data` 的 20000 blob **删除**;头条总数、「命中 N 岗」、「显示更多」三处数字**自洽**。
- 目标:个性化三功能(匹配视图 / 「今日 X 高匹配」FOMO / 推荐板)**功能不降级**,走独立端点。
- 非目标:改默认排序(发布时间最新在前不动)、改筛选项/列、改视觉布局;keyset 游标(本期用 OFFSET);
  分类中文标签的全文搜索(本期核心列,后续补);match() 规则本身(lib/match.ts 不动)。

## 3. 方案总览(架构变化)

```
【改前】SSR 50 行 ──► 客户端拉 20000 blob ──► 前端 filter/sort/paginate(全在浏览器)
【改后】SSR 第0页 50 行 + 维度 + 总数 ──► 筛选/搜索/翻页每次打 /api/jobs(SQL) ──► 前端只渲当前累计页
```

- **单一查询真相**:筛选→SQL 的映射只写一处 `lib/jobsQuery.ts`(WHERE 拼装 + 分页 + count),
  SSR 首屏、`/api/jobs`、个性化端点全复用它,不各写各的(承接 CLAUDE.md「清洗/口径下沉数据层」)。
- **维度独立**:省/市/区/NOC 分类下拉选项改为**只**读维度表(provinces/cities/districts/noc_categories),
  不再从 blob 派生(blob 没了)。维度小(合计 <500KB),随首屏或单独端点一次拉。

## 4. API 契约

### 4.1 `GET /api/jobs`(浏览/筛选/搜索/分页 —— 非个性化)

Query 参数(全部可选,缺省=不筛):
`page`(默认 0)· `q`(搜索串)· `prov` `city` `district` · `broad` `mid` `fine` `teer`
· `source` `acc` `pnp`(yes/no)`aip`(yes/no)`status` `origin` · `score` `sal` `vs` `emp` · `direct`(1)
· `sort`(列 key,默认 datePosted)`dir`(asc/desc)

返回:
```json
{ "rows": [JobRow...], "total": 12345, "page": 0, "pageSize": 50, "updatedAt": "..." }
```
- `pageSize=50`(= 现 PAGE_ROWS);`rows`=当前页 50 行(带 match,若 profileOk);
- `total`=**同 WHERE 的 count(*)** → 头条命中数 + 「还有 N」全用它算,天然自洽;
- `updatedAt`=`SELECT max(last_seen) FROM jobs`(全局,一次)。

### 4.2 个性化三端点(view/count/rec)

| 端点 | 用途 | 做法 |
|---|---|---|
| `GET /api/jobs?view=match&page=` | 匹配视图 | SQL 预筛候选(`province = ANY(targets) OR noc = ANY(profileNocs) OR pnp_eligible OR ee_category<>''`)→ TS 跑 `match()` → 留 high/mid → 按 match rank+日期排 → 分页 |
| `GET /api/jobs/match-count` | 「今日 X 高匹配」FOMO | 同预筛候选集跑 match() → 计 high/mid 数(不返行) |
| `GET /api/jobs/recs?prov=&broad=&sal=` | 推荐板前 3 + 总数 | `WHERE prov/broad/sal ORDER BY score DESC LIMIT 3` + 同 WHERE count(「看全部 N」) |

- 预筛用 SQL 把候选从 3 万压到几百~几千(用户 target 省 + profile NOC),再 TS `match()`——
  避免全表跑 TS,又保证 match() 规则单一来源不复制到 SQL。
- 未建档 / 未登录 → 这三端点直接空(与现 `profileOk` 门槛一致)。

## 5. SQL 层(`lib/jobsQuery.ts`,新增)

### 5.1 筛选 → WHERE 映射(16 项,参数化占位符防注入)

| 前端筛选 | WHERE 片段 |
|---|---|
| prov/city/district | `j.province = $` / `j.city = $` / `j.district = $` |
| broad/mid/fine | `j.broad = $` / `j.mid = $` / `j.fine = $` |
| teer | `TEER n`→`j.teer = n`;`未分类`→`j.teer IS NULL` |
| source | `j.source_label = $` |
| acc | `j.accessibility = $` |
| pnp | yes→`j.pnp_eligible`;no→`NOT j.pnp_eligible AND j.province <> 'QC'` |
| aip | `j.aip = $(bool)` |
| status | `j.status = $`(缺省不限;注意现默认展示含 open/closed,保持) |
| origin | `j.origin = $` |
| score | 档位→`j.score >= lo [AND < hi]`(复用现 okScore 边界) |
| sal | 档位→`j.salary_annual` 区间(复用现 okSal 边界:ge100/80/60/u60) |
| vs | `j.salary_annual / NULLIF(j.wage_med_annual,0)` 与 1.0/1.2 比(复用 okVs) |
| emp | full/part→`j.employment_hours=$`;gig→`(employment_hours='part' OR employment_term IN ('casual','seasonal'))` |
| direct(仅雇主直发) | `(COALESCE(j.apply_url,'') NOT ILIKE '%jobbank.gc.ca%' OR COALESCE(j.source,'')='Job Bank')`——旧 isDirect=Job Bank 渠道只留一手直发、ATS 一律直发(**不是** is_agency) |
| q(搜索) | `(j.title ILIKE $p OR c.name ILIKE $p OR j.city ILIKE $p OR j.province ILIKE $p OR j.district ILIKE $p OR j.noc ILIKE $p OR j.source_label ILIKE $p)`,`$p = '%'||term||'%'` |

- 边界常量(score/sal/vs 档位)从前端 `okScore/okSal/okVs` 抽到 **`lib/jobFilters.ts` 共享**,前后端同一份(不让 SQL 和 TS 各写一套阈值)。

### 5.2 分页 / 排序 / count

- 排序:`ORDER BY <sortCol> <dir>, j.score DESC NULLS LAST, j.id DESC`(默认 sortCol=date_posted desc,与现一致)。
- 分页:`LIMIT 50 OFFSET page*50`。
- count:同 WHERE 跑 `SELECT count(*)`(与行查询并发 `Promise.all`)。深页 OFFSET 慢在本站不触发(用户不深翻)。

## 6. 前端改造(`JobsTable.tsx`)

- **删**:`/api/jobs-data` 拉 blob 的 effect;`hays`/`sorted`/`rows` 三个客户端计算 useMemo;`searchHay`。
- **改**:筛选/搜索/排序/matchView state 变化 → `useEffect` debounce(搜索 ~300ms,下拉即时)拼 query 打 `/api/jobs`,
  `setRows(resp.rows)`、`setTotal(resp.total)`;「显示更多」= `page+1` 拉下一页 **append**;
  筛选/排序变 → `page=0` 重置替换。
- **状态**:加 `loading`(请求中盖轻量态,承接 B.4 遮罩风格)、`total`(替代 `rows.length` 口径)。
- **数字自洽**:头条命中=`total`;「显示更多(还有 N)」=`total - rows.length`;不再有 20000 天花板。
- **维度下拉**:确保 `dims.provinces/cities/districts` 恒有值(首屏或维度端点),删掉 `jobs.filter(...)` 派生兜底路径。
- 竞态:请求带自增 `reqId`,晚到的旧响应丢弃(AbortController + seq 双保险)。

## 7. SSR 首屏(`jobs/page.tsx`)

- 首屏仍 SSR 第 0 页 50 行(用 `lib/jobsQuery.ts` 无筛选 page0)+ 维度 + `total`(count)+ `updatedAt`。
- URL 有筛选参数(如 `?prov=ON`)→ SSR 直接按参数出对的首屏(SEO/分享/回流一致)。
- 删 `deferFull`(不再有「后台补全量」概念)。

## 8. 影响面与风险

- 影响文件:`lib/jobsList.ts`(重构为 jobsQuery)、`api/jobs-data`(拆/废)、新增 `api/jobs`+个性化端点、
  `jobs/page.tsx`、`JobsTable.tsx`(大改)、抽 `lib/jobFilters.ts`。
- 风险①:**筛选口径回归**——16 项逐项要与现客户端行为逐一比对(尤其 pnp=no 的 QC 排除、gig 组合、vs 比值 NULL)。
  缓解:验收逐项对照现网结果计数。
- 风险②:搜索**降覆盖**——分类中文标签/「TEER n」字符串本期不搜。缓解:核心列覆盖 95%,文档标注 known-gap,后续补。
- 风险③:每次筛选一跳网络,手感变「有延迟」。缓解:debounce + loading 态 + 下拉即时;DB 有 province/city/noc 索引(需确认/补索引)。
- 风险④:匹配视图/推荐的候选预筛若过窄会漏。缓解:预筛条件取并集从宽,TS 侧再精筛。
