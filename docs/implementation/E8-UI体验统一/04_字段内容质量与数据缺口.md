## 3. 高质量目标 · 逐字段(现状 → 目标 → 缺口→抓取)

### 3.1 公司(company)—— 事实段空,AI 段已有实时 grounding
- **现状(修正)**:点「公司信息」事实段=官网(24%)+ 行业(空)+ 简介(空)+ 在榜职位;**AI 段已由 E6-03 实时 web_fetch 抓官网做 grounded 分析**(2026-07-05 生产验证过,能抓出官网当前产品/slogan)。所以不是全空壳——空的是「事实段」(DB 的 description/sectors),AI 段有真材料。
- **矛盾点**:E6-03 是**运行时抓、不落库**(零 ETL)。好处=零维护、总是最新;代价=每次开弹窗重抓(慢+费 token)、事实段仍空、无法用于列表/筛选/SEO。
- **两条路(§7 待用户拍板)**:
  - **A. 维持现状 + 打磨事实段**:AI 段够用,只把事实段的空行诚实降级(无简介不显示「未抓到」长文,已做),行业若抓到顺手落库。零新 ETL。
  - **B. 落库增强(真「去抓」)**:新增 `enrich_companies` ETL——按官网 URL(24% 直接有,其余 web_search 兜底找)抓首页 → 存 description/sectors/website。事实段从 0.1% 提到 ≥60%;AI 改吃库里事实(更快更省);可用于列表/SEO。代价=新 ETL 脚本+维护+抓取量(~1 万公司,分批)。
- **护栏**:落库的简介标注抓取源+日期;AI 生成的推断标「据公开信息,可能有误」(现有先例)。

### 3.2 地点(address / district)
- **现状**:点「地址」过半空;「区」37%。
- **目标**:能精确到街道就给地图针;不能就诚实到市级,不猜。
- **缺口→抓取**:address 缺失多是 Job Bank 源本身只给到市。**不硬抓**(政府源没有的编不出来)——改为**诚实降级**:无街址时地图链接落市级(已在 E8「地点按层级」做了一半),弹窗不显示空地址行(已做)。district 靠 GeoNames 邮编反查已有 37%,提升需邮编覆盖率,低优先。
- **判定:此项不新增抓取,靠「诚实降级」达标**(符合规范哲学「缺数据明说」)。

### 3.3 职位 JD(title)
- **现状**:78% 有正文,版式已按 Job Bank/Indeed 双格式整形(07-06 完成)。
- **目标**:剩 22% 无正文的,弹窗明说「该岗位来源未提供正文,点原帖查看」+ 官方原帖链接(已有降级文案 act.noText)。
- **缺口→抓取**:22% 多是 ATS 岗抓取时正文在 JS 渲染后 → 归 E1-03 抓取增强,非本批。达标靠现有降级。

### 3.4 薪资五字段 / 分类五字段 / 时间四字段 / 来源三字段
- 数据 99% 齐、排版已在 E8 逐字段对齐(07-06)、来源各归其源(07-06)。**只差按规范补口径注的措辞打磨**,无数据缺口。
- 时间字段已加「已挂 N 天」新鲜度(P0 痛点项,737ecd2)。

### 3.5 PNP / EE / AIP / LMIA(身份信号族)
- 判定完整、清单带官方链接(PNP 通道名挂政策页,737ecd2)。**高质量已达标**,是付费头牌内容。

## 8. 立项:web_search 兜底找官网(D2,待拍板)

**问题**:enrich 只能抓「已有官网 URL」的公司(2,501/10,432 = 24%,URL 来自 ATS 名录)。其余 ~7,900 家
(几乎全是 Job Bank 岗的雇主)没有 URL 可抓 → **公司简介覆盖率封顶 ~24%**。突破需先「找到官网」。

**方案(最小实现)**:enrich_companies.py 加一步「找官网」——
1. 输入:无 website 的公司(按在招岗数降序,岗多的先做,价值密度高);
2. 查询:搜索引擎查 `"<公司名>" <省> official site`(候选源:DuckDuckGo HTML 无 key / Bing API 有免费额度);
3. **护栏(信任边界,不上砧板)**:取首页候选域名后做「域名 ↔ 公司名」相似度校验(归一化公司名 token 在域名/官网 `<title>`/og:site_name 中命中才算),不达阈值=放弃留空,**宁缺勿错**——错认官网→写错简介,比没有简介伤害大;
4. 命中:存 `website`(标 `found: searched`,与名录来的区分)→ 下轮 enrich 正常抓简介;
5. 节奏:与现有 enrich 同轮同批(每轮 300 家),零新基建。

**诚实前提**:长尾小雇主(餐馆/农场/小工程队)很多真没官网——找不到=留空,弹窗照旧不留孤儿行。
预估最终覆盖:24% → 50-65%(按域名校验通过率经验值,跑首批 300 家后校准)。

**成本/风险**:搜索源反爬(DDG HTML 偶发限流,失败即跳过不重试);错认风险由 3 的护栏兜住;
代码量 ~60 行(现有脚本内加函数),无新依赖(httpx 已有)。

**拍板(2026-07-06 深夜)**:做;DuckDuckGo;标小字。用户点名「用自己的爬虫先抓」→ 方案升级为**三级阶梯**:

| 级 | 手段 | 置信 | 产能 | 状态 |
|---|---|---|---|---|
| ① | **JD 正文域名线索**(已抓 JD 里的邮箱/链接域名,雇主自己写的) | 高 | 一次性 ~56 家(实测) | ✅ 已实现 |
| ② | **补抓 Job Bank 申请邮箱**(详情页「How to apply」在按钮后,现爬虫没点;`hr@company.ca`→官网) | 最高(雇主自报) | ~8,000 家详情补抓,礼貌速率跑几天;gmail 落空,预估命中 20-40% | ☐ 待排期(爬虫改动大) |
| ③ | **DDG HTML 搜索**(每轮 --find-limit 60 家,岗多的先搜) | 中(护栏兜底) | 逐轮累积 | ✅ 已实现 |

**①③ 实施记录(2026-07-06 深夜)**:
- enrich_companies.py 新增 `find_websites()` 阶梯 + `guard_match()` 护栏(域名 token 命中 ≥2/单词全中/首字母缩写;
  域名对不上再查首页 `<title>`/og:site_name;聚合站/通用邮箱域黑名单;**对不上=nosite 冷却 90 天,不猜**)。
- `found: jd|searched` 跟记录走 → 09 合并成 companies.websiteSource → seed(website_source 列)→ jobsList SQL
  (officialUrl 帖级优先、公司级兜底)→ 公司弹窗小字 `fact.siteSearched`(仅 searched 标;jd 是雇主自己写的不标)。
- 小样本实测:JD 线索 +56 家全部合理(tcfoods.ca 走标题复核、ocsb.ca 走缩写规则);DDG 5 搜 2 中;
  法语带音符公司名是已知弱点(护栏正确放弃)。mart websiteSource 58 行验证通过。
