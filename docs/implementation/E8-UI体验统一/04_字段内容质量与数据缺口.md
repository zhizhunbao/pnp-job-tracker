## 1. 整体目标

弹窗里每个字段都做到「有实质内容 + 排版对齐规范 + 缺数据诚实/去补」。付费逻辑:用户点开字段看到的是**别处查不到、我们代查代整理**的结论,而不是一行空值或「—」。

## 3. 高质量目标 · 逐字段(现状 → 目标 → 缺口→抓取)

### 3.1 公司(company)—— 事实段空,AI 段已有实时 grounding
- **现状(修正)**:点「公司信息」事实段=官网(24%)+ 行业(空)+ 简介(空)+ 在榜职位;**AI 段已由 E6-03 实时 web_fetch 抓官网做 grounded 分析**(2026-07-05 生产验证过,能抓出官网当前产品/slogan)。所以不是全空壳——空的是「事实段」(DB 的 description/sectors),AI 段有真材料。
- **矛盾点**:E6-03 是**运行时抓、不落库**(零 ETL)。好处=零维护、总是最新;代价=每次开弹窗重抓(慢+费 token)、事实段仍空、无法用于列表/筛选/SEO。
- **两条路(§7 待用户拍板)**:
  - **A. 维持现状 + 打磨事实段**:AI 段够用,只把事实段的空行诚实降级(无简介不显示「未抓到」长文,已做),行业若抓到顺手落库。零新 ETL。
  - **B. 落库增强(真「去抓」)**:新增 `enrich_companies` ETL——按官网 URL(24% 直接有,其余 web_search 兜底找)抓首页 → 存 description/sectors/website。事实段从 0.1% 提到 ≥60%;AI 改吃库里事实(更快更省);可用于列表/SEO。代价=新 ETL 脚本+维护+抓取量(~1 万公司,分批)。
- **护栏**:落库的简介标注抓取源+日期;AI 生成的推断标「据公开信息,可能有误」(现有先例)。

### 3.2 地点(address / district)
- **现状**:点「地址」过半空;「区」37%。
- **目标**:能精确到街道就给地图针;不能就诚实到市级,不猜。
- **缺口→抓取**:address 缺失多是 Job Bank 源本身只给到市。**不硬抓**(政府源没有的编不出来)——改为**诚实降级**:无街址时地图链接落市级(已在 E8「地点按层级」做了一半),弹窗不显示空地址行(已做)。district 靠 GeoNames 邮编反查已有 37%,提升需邮编覆盖率,低优先。
- **判定:此项不新增抓取,靠「诚实降级」达标**(符合规范哲学「缺数据明说」)。

### 3.3 职位 JD(title)
- **现状**:78% 有正文,版式已按 Job Bank/Indeed 双格式整形(07-06 完成)。
- **目标**:剩 22% 无正文的,弹窗明说「该岗位来源未提供正文,点原帖查看」+ 官方原帖链接(已有降级文案 act.noText)。
- **缺口→抓取**:22% 多是 ATS 岗抓取时正文在 JS 渲染后 → 归 E1-03 抓取增强,非本批。达标靠现有降级。

### 3.4 薪资五字段 / 分类五字段 / 时间四字段 / 来源三字段
- 数据 99% 齐、排版已在 E8 逐字段对齐(07-06)、来源各归其源(07-06)。**只差按规范补口径注的措辞打磨**,无数据缺口。
- 时间字段已加「已挂 N 天」新鲜度(P0 痛点项,737ecd2)。

### 3.5 PNP / EE / AIP / LMIA(身份信号族)
- 判定完整、清单带官方链接(PNP 通道名挂政策页,737ecd2)。**高质量已达标**,是付费头牌内容。
