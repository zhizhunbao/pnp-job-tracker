# E8-11 弹框页面同源化(以弹框为准,一域一源)

> 缘起:2026-07-22 Frank 拍板「弹框和具体页面可以改成一个来源吗」→「以弹框为准,分类/公司/job 还有其他弹框都这么设计」。
> 直接痛点:#188/#190/#192 三次「同一逻辑弹框页面各改一遍」;简介阈值(200 vs 120)、在招职位行(富/简)、额度口径(收/免)已实测漂移。
> 本件=设计文档,获批后按批次实施(B1 公司先行)。

## 1. 架构原则(整件的宪法)

- **一域一 Body 组件**:每个内容域(公司/职位/分类/移民)一个渲染骨架组件,放 JobsTable.tsx 里 export(循 #120「只加 export 不搬代码」铁律)。弹框和页面渲同一个 Body,**改一处两边生效**。
- **一域一数据源**:同一域两处消费同一份数据结构(页面 SSR 直查,弹框走轻端点返回同一形状);字段拼凑、双阈值退役。
- **两壳各留各的**:弹框壳=浮层+三钮行(对照/AI 速读/完整页);页面壳=面包屑+H1+JSON-LD+SEO。壳只管包,不管内容。
- **以弹框为准 = 排版基准**:JD 扁平风(节头加粗同字号、正文缩进 14、无分隔线、13px/1.75,FLAT_* 三常量)是全站内容区的排版语言;**页面卡片风退役,跟弹框换扁平**(#187「先只改弹框」的另一半在此收口)。
- **页面可加深不可少**:页面 = Body + 页面独有深块(SEO 命脉不砍,#175 拍板)。弹框内容 ⊆ 页面内容。

## 3. B1 公司域细化(先行批)

- **数据**:新端点 `/api/company`(POST name 或 slug)→ 复用 `fetchCompanyBySlug` 返回 `CompanyDetail`(+similar);弹框改吃它,job 行字段拼凑与 `/api/companyinfo`/`/api/scoredetail` 双 fetch 退役(companyinfo 仍是懒查引擎,由端点服务端内部触发,K 懒探索不变)。
- **骨架** `CompanyBody`:头部身份 → 担保记录深块(有记录才出)→ 公司简介(desc/AI 互斥,阈值统一 120)→ 在招职位(富行:NOC 对照+薪资+通道档)→ 相似雇主 → 雇主信号(殿后,#192)。
- **弹框增益**:担保深块+相似雇主+富职位行进弹框(parity);**页面换扁平**(sec 卡壳退役)。

## 4. 连带拍板(随 B1 生效)

- **额度统一免费**:公司域数据全免费(粗筛信号;付费核=AI 生成+档案匹配)。`/api/company` 不过 freeGate;**#189 公司弹框页眉额度注随之退役**(不烧池无可显;职位弹框的额度注不受影响)。
- **#192 三改**(雇主信号改名/殿后/免责注摘)是 Body 的既定顺序,同源化后自动两边一致。

## 5. 红线

- **分类弹框与 job(JD)弹框功能不碰**(Frank 2026-07-22 开工令附带:「这两个现在做的我很满意」)——B1 只动公司域;与其共用的组件只加 export 不改行为;B2/B3 动它们前必须单独拿 Frank 批。
- SEO 页面内容只增不减(#175);sitemap/JSON-LD/面包屑不动。
- 懒查铁律不破:AI 简介仍首开才查、永久缓存([[lazy-first-company-data]])。
- 每批验证法:改 Body 任一节 → 双视口(375/桌面)双上下文(弹框/页面)各截一张,四图同变才算过。
