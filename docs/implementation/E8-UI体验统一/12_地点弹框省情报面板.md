# E8-12 地点弹框(省情报面板)

> 状态:**实施中**(2026-07-23 Frank 拍板,效果图当日定稿)
> 前置:E8-10 弹框三合一(FieldGroup 机制)、E12-07 省难度指数(stats.difficulty)、E12-06 省新闻、E6-04 省抽选。
> 效果图:两列五卡版(2026-07-23 session 内 Playwright 截图定稿)。

## 1. 为什么做

Frank(2026-07-23):「省 市 地区 也需要像分类一样弹框吧」「点击文字的时候才跳转 map,点击框的时候应该弹框」「省需要包含这个省的相关数据啊,比如省相关的政策啊、人数啊、工签、临时居民、PR 数量之类的」「省可以包含很多信息啊,不同的卡片」。

E8-10 曾把省/市收成「图」档(点了直连地图)——当时删的是「Ontario 是加拿大的一个省」这种一句话 AI 解释。现在装的是**真数据**(IRCC 官方体量数、难度指数、抽选、公告),不是解释,与 #176「入口语义=内容」一致:点地点看地点(这地方移民生态什么样)。

## 2. 拍板结果(2026-07-23,Frank)

- **交互**:省/市/区 三列,**点文字 → 跳地图,点格子空白 → 开弹框**。地址列不动(仍整格直连地图)。
  **手机卡**(次日补拍「手机卡片呢?」):卡上没有「格子」→ 地点文字=开弹框(长按/新开标签仍地图,#131 职位名同款);弹框内地点卡文字跳地图。
- **弹框**:第 4 个 FieldGroup `location`,五张卡,**竖排单列**+顶部「打开完整页 ↗」钮(→ /stats/[prov] 地区统计页)。
  沿革:效果图期 Frank 要过两列(「尽量左右两列」),上线当天看实物推翻——「别横着排列,其他的弹框都是竖着排列的」;中文对照/AI 速读钮不适用(内容已本地化、零 AI 设计),钮行只出打开完整页。
- **文案**:TFWP/IMP 用户看不懂 → 人话名主文案 + 代码灰注(「工签:雇主担保类 <TFWP>」「工签:开放及豁免类 <IMP,含毕业工签>」)。

## 3. 五张卡

| 卡 | 内容 | 数据源 | 状态 |
|---|---|---|---|
| ① 地点 | 国/省/市/区/地址,点进来的行高亮(同分类卡①);有值行文字=地图链接 | job 行字段 | 前端即有 |
| ② 移民难度 | 档位+竞争比+配额同比+抽选活跃(E12-07 同源) | stats 表 broad='all' 行 difficulty | 已入库 |
| ③ 该省移民体量 | 学签/工签(TFWP/IMP)在库、PNP 配额(今明两年)、PR 登陆(PNP 类) | data/raw/ircc/* → **本次新入库** | 本次 |
| ④ 本省最近抽选 | PnpDrawsBlock 复用 | pnp_draws 维度(dims 已加载) | 已有 |
| ⑤ 本省最新公告 | NewsLatestBlock 复用 | news 维度(dims 已加载) | 已有 |

QC 特判:卡②④不适用(魁省不走 PNP,沿用 isQc 手法出说明行);卡③照出(IRCC 数含 QC)。

**E8-12b 三级分流(2026-07-23 Frank 追拍「点省看省,点市看市」+「点区看区」)**:卡①地点三级共用(点哪级哪行高亮),其余卡按点击层级分流,互不泄漏:

| 层级 | 卡 | 内容 | 数据源 |
|---|---|---|---|
| 市 | 该市就业市场 | 在招/近 7 日新增/帖面中位年薪/热门方向 top3 | `/api/city` 现算 jobs 聚合(懒查询,2,346 城不预计算) |
| 市 | 可申毕业工签的院校 | 数量 + 前 4 所(公立灰注) | dli 维度(E12-03,city 列现成) |
| 市 | AIP 指定雇主 | N 家 + 雇主名录链(仅大西洋四省城市有数) | designated_employers(location 匹配) |
| 区 | 该区就业市场 | 同四指标,区口径 | 同 `/api/city`(district 参数) |
| 区 | 本区主要雇主 | top 4 按在招数,名可点进公司页 | jobs⋈companies 现算 |

区列点开但该岗无区值 → 退回市级(不出空面板)。

1. `09_build_mart.py`:provinces.json 每省挂 `info` JSON —— 从 `data/raw/ircc/temp_residents.json`(学签/TFWP/IMP byProv + 年份)、`pnp_admissions.json`(PR 登陆+年份)、`pnp_allocations.json`(今明两年配额)拼装;缺省留空不瞎猜。

   ⚠️ **DB_PUSH 雷区(2026-07-23 实测)**:`DB_PUSH=1` 起 dev 时 drizzle 会威胁**删除** `jobs.eligibility_flag/quote`、`companies.lmia_positions_skilled` 三列(它们是 docs/sql 手写列,故意不在 Payload schema 里,GAP1③/B4-02 先例)——**那个 y/N 提示绝不能答 y**(41k 行真数据)。凡涉及 docs/sql 列共存的库,加列一律走 docs/sql 手写 SQL,不再用 DB_PUSH。
