## 0. 两条核心原则(最重要)

**原则 A —— 字段由功能驱动,不照抄论坛。** 一亩三分地是社区论坛(积分/大米/马甲/签名/声望),我们是**移民求职工具**。
借它的是「头像 + 干净账户下拉 + 基本身份」的 UI 范式;**不借**论坛社交那套(积分体系/马甲/个人签名/发帖数)。
每个字段都要能回答「它驱动哪个功能」——驱动不了的现在就不收(Ponytail/YAGNI:最好的字段是你没加的字段)。

**原则 B —— 通俗易懂 + 低门槛不吓人(2026-07-17 Frank 强调)。** 用户**不懂** NOC / TEER / CLB / CRS 这些官方黑话。
- **术语翻译**:界面**一律不问官方码/术语**,用大白话问(「你想做什么工作?」而非「你的 NOC 代码」),
  **幕后**把答案映射成结构化码(职位名→NOC、英语水平→CLB…)。这与项目「清洗下沉数据层」一致:**翻译是数据层的活,不甩给用户**。
- **不搞敏感长表单**:不要护照号/SIN/生日/确切考分强制、不做移民官式问卷——一沾「敏感 + 官方感」用户就跑。
- **但要问到够**:匹配需要最小信号(想做什么工作 / 想去哪 / 英语大概水平 / 现在什么身份)。
- **能选就别打字(2026-07-17 Frank 强调,原则 B 的核心手法)**:凡是能下拉/点选的**一律不让用户打字**——打字=门槛+流失。
  - 首选:单选按钮 / 多选 chips / 下拉;
  - 必须自由输入的(如职位):给**推荐选项/预置答案**让他直接点(热门职业 chips + 分类下钻),**搜索只作兜底**,**类似 Claude 给你推荐回答那样**;
  - 连数字也别让填:PGWP 剩余、CRS 分 → 给**区间选项**(<6月 / 6–12月 / 1–2年 / 不清楚),不让敲具体数。
- **其它手法**:① 一步一两个问题不是长表单;② 每项**可跳过**;③ **价值前置**(「填了立刻看到匹配度」);④ 进度可见(「3 步搞定」);
  ⑤ 常见答案可**预选/推荐**,用户确认即可。矛盾的解法=**大白话问最小集 + 零打字点选 + 幕后翻译 + 可跳过**。

## 1. 背景与目标

- 现状:登录只有**邮箱+密码**(Payload 自带 auth,httpOnly cookie);用户信息=邮箱 + 移民档案(E5-00:NOC/CLB/CRS/目标省/PGWP)。
  无头像、无昵称、无社交登录、无求职意向结构化字段;账户页有 sidebar,但顶栏「我的账户」下拉不完善(待办 B.6)。
- 目标:① 降注册门槛(Google 一键 + 头像/昵称让账户「像个账户」);② 把用户的**求职+移民意向**结构化,喂给匹配/顾问/未来的简历与择校;
  ③ 账户下拉补齐求职向条目。
- 非目标:论坛化(积分/马甲/公开主页/关注)、一次性问全部信息、为还不存在的功能预收字段。

## 2.5 用户分型(2026-07-17 Frank 强调:不同的人 → 不同选项/状态/路径/推荐)

**用户不是一类人。** 国内海外直申 vs 在加留学生,移民路径天差地别。所以 `currentStatus`(§3.3)**不只是一个字段,是整个 onboarding 与体验的分叉点**——
选了哪类人,后面**问什么、给什么岗、强调哪条路径**全都不同。这是「一步分型 → 后续按型走」的设计,而非给所有人同一张表。

| 分型 | 状态 | 主移民路径 | 后续问什么(分叉) | 产品强调/给什么 |
|---|---|---|---|---|
| **A 海外直申** | 还在国内/海外,没来加 | EE 联邦(FSW,凭 CRS,不需 offer/加拿大经历)+ 部分海外友好省提名 + 远程拿 offer→工签 | 目标职业 · 英语水平 · **算过 EE 分吗**(帮估)· 想去哪个省 | EE 估分、紧缺职业类别、**海外可申的省通道**、能远程/担保的雇主 |
| **B 在加留学** | 正在加读书 | 毕业→PGWP→加拿大经验→CEC/省提名 | **什么时候毕业** · 目标省 ·(可选)读什么专业 | 毕业后职业匹配、PGWP 时间线、**目标省毕业生 PNP 通道**、雇主 offer 岗 |
| **C 在加工签在职** | 有工签在工作 | 攒经验→CEC/省提名 | 现在做什么 · 目标省 | 更好的岗、**当前岗够不够 PNP**、雇主担保(外劳)记录 |
| **D 在加找工作** | PGWP/毕业待业 | 找 PNP-track 岗→省提名 | 目标职业 · 英语 · 目标省 | **雇主直发岗**、命中省清单的岗、匹配度 |
| **E 已 PR / 纯找工** | 已 PR 或不需移民 | 纯求职 | 目标职业 · 地点 | 就是找好工作,**移民信号弱化**(别硬塞 PNP) |

- **分型即第一问**(单选、大白话、点选):「你现在是什么情况?」→ 从此按型走(问题分叉、路径话术分叉、岗位推荐权重分叉)。
- 分型可改(处境会变:留学生毕业→变 D→拿到工签→变 C);改分型 → 后续引导与推荐随之变。
- 与现有逻辑衔接:匹配层(match.ts)/顾问 grounding 已按档案算;分型给它们补「这人处于哪条路径」的语境
  (顾问身份红线正需要:读者是留学生还是海外直申,话术完全不同)。

## 3. 用户信息模型(三层)

分三层:**账户身份层**(是谁)/ **移民档案层**(已有,匹配用)/ **求职意向层**(新增,漏斗用)。
每字段标注:驱动功能 · 收集时机 · 是否已有。收集时机遵循渐进式——注册最小,用到才引导填。

### 3.1 账户身份层(新增为主)

| 字段 | 类型 | 必填 | 驱动功能 | 收集时机 | 现状 |
|---|---|---|---|---|---|
| email | email | ✅ | 登录/通知 | 注册 | 已有 |
| loginProvider | text | — | 区分邮箱/Google/微信;账户页显示、绑定 | 注册自动 | 新增 |
| displayName 昵称 | text | — | 问候语、未来评论/社区署名 | 注册后可选(默认取邮箱前缀/Google 名) | 新增 |
| avatar 头像 | media/URL | — | 账户辨识、亲切感 | 可选(默认首字母块) | 新增 |
| locale 语言 | text | — | 三语界面/邮件语言(现在只存 localStorage,没落用户) | 首次选择 | 半有(未落库) |

### 3.2 移民档案层(已有 E5-00,不动结构,只在 UI 引导时机上纳入渐进式)

`nocCodes`(目标职业/NOC)· `clb` · `crs` · `targetProvinces`(目标省)· `pgwpMonthsLeft` · `profileUpdatedAt`
→ 驱动:岗位匹配(match.ts)、匹配版邮件提醒、顾问档案感知。**「选什么工作」「选哪个省份」这两个诉求这层已覆盖**(NOC=职业、targetProvinces=省)。

> ⚠️ **连带整改(原则 B)**:现有 ProfileForm **违反通俗易懂**——NOC 搜索结果 chip 直接显示码(「31103 · registered nurse」)、
> CLB/CRS 直接标术语让用户填分。**结构(字段)不变,UI 问法要改**:职位搜索只显示职位名藏码、CLB 改问「英语水平」、
> CRS 改「可选/帮你估」。这是 P1/P3 的实打实工作项(§9),不是纯新增。

### 3.3 求职意向层(新增 —— Frank 点名的「工作/学校/省份」里,学校+身份是新的)

| 字段 | 类型 | 驱动功能 | 收集时机 | 说明 |
|---|---|---|---|---|
| 当前身份 currentStatus | select = **§2.5 分型 A–E** | **onboarding 分叉点**、顾问 grounding(读者身份红线)、匹配 caveat、路径话术 | **第一问** | 见 §2.5;不只字段,是分叉锚 |
| 目标学校/项目 targetSchool | text/结构化 | **择校 + 证书路径(F3/F4)** | 该功能上线时才收 | ⚠️ 功能未立项,先设计字段、**暂不收**(YAGNI) |
| 期望工作地 | = targetProvinces | 匹配/推荐 | 已在移民层 | 不重复设,复用省字段(可加城市粒度) |
| 目标职业方向 | = nocCodes(+可选口语行业) | 匹配/推荐 | 已在移民层 | 口语行业仅作 onboarding 引导,落库仍归一到 NOC |
| 经验年限 / 期望薪资 / 时间线 | number/text | 简历环、匹配精修(产品 v3 六环) | 对应功能上线时 | ⚠️ 同上,先设计不预收 |

> 关键取舍:**「目标学校」「经验/薪资/时间线」等对应的功能(择校 F3/F4、简历环)还没立项 → 现在只在模型里设计,不真收**
> (收了没功能消费 = 死字段,违 Ponytail)。当前真正落库的新字段=身份层(loginProvider/displayName/avatar/locale)+ currentStatus。

### 3.4 术语翻译表(界面怎么问 vs 幕后存什么)—— 落实原则 B

| 幕后字段(存) | ❌ 别这样问 | ✅ 大白话问 | 🖱 零打字输入法 | 幕后映射 |
|---|---|---|---|---|
| nocCodes | 「你的 NOC 代码」 | 「你想做什么工作?」 | **热门职业 chips**(护士/软件/会计/厨师/司机/PSW…)+ 分类下钻点选;搜索仅兜底 | 选中项→NOC,**只显示职位名不露码** |
| teer | 「你的 TEER 等级」 | (**根本不问**) | — | NOC→TEER 内部算 |
| clb | 「你的 CLB 等级」 | 「英语大概什么水平?」 | **单选按钮**:初级/中级/流利/考过雅思思培(选分段) | 档位→CLB |
| crs | 「你的 CRS 分数」 | 「算过 EE 分吗?」 | **单选**:没算(帮你估)/ 算过→**分数区间下拉**,不敲具体数 | 可空;后续帮估 |
| targetProvinces | 「目标省份代码」 | 「你想去哪个省?」 | **省份多选 chips**(现成) | 省名→省码 |
| pgwpMonthsLeft | 「PGWP 剩余月数」 | 「工签还剩多久?」 | **区间单选**:<6月/6–12月/1–2年/不清楚 | 区间→值,可空 |
| currentStatus | 「你的移民身份类别」 | 「你现在是什么情况?」 | **单选按钮**:留学生/工签在职/找工作/还在国内/已工作 | 直接存 |

> 一句话:**用户说人话、点两下就完;系统存代码**。能选就不打字,要打字就给推荐选项(类似 Claude 推荐回答);
> 官方码/术语全在数据层翻译,界面零黑话、零敏感官方问卷、零(或最少)键盘。

## 4. 社交登录方案(拍板:Google+邮箱先上,微信缓)

| 方式 | 状态 | 技术路径 | 依赖/门槛 |
|---|---|---|---|
| 邮箱+密码 | ✅ 现有 | Payload 自带 auth | 无 |
| **Google** | 🎯 近期 | OAuth 2.0 → 换 Payload 会话(自建 /api/auth/google/callback 或 Payload OAuth 插件);首登建用户、存 loginProvider+avatar+name | Google Cloud OAuth 客户端(Frank 亲手在控制台建凭据,助手接线);回调域 offer2pr.com |
| **微信** | ⏸ 缓(backlog) | 微信开放平台「网站应用」扫码登录 | **需企业资质审核**(已知门槛,周级);受众终态目标,资质下来即接 |
| Apple(可选) | 未定 | iOS 场景才需 | 暂不 |

- 账户可**多登录方式绑定同一邮箱**(Google 邮箱 == 已注册邮箱 → 绑定而非重复建号),避免一人两号。
- 隐私:OAuth 只取 email + 基本 profile(name/avatar URL);不取通讯录/好友;头像可存 URL 或落我方存储(见 §5)。凭据只在服务端。

## 5. 头像方案

- **默认**:首字母色块(邮箱/昵称首字母 + 稳定色),零上传零存储成本——多数用户不会传。
- **上传**:Payload media(→ Supabase Storage);限图片、限尺寸、裁方形;或直接用 OAuth 带回的头像 URL(省存储)。
- **建议**:v1 = 首字母默认 + OAuth 头像 URL(不做上传);上传留后续(用得上再加,Ponytail)。

## 6. 账户下拉布局(承接待办 B.6,去论坛化)

顶栏「我的账户」下拉(登录态),参考 1p3a 布局范式但换求职向条目:

```
┌─────────────────────────┐
│ [头像] 昵称 / 邮箱        │  ← 头部:头像+身份+Pro 标
│  Free / Pro(到期日)     │
├─────────────────────────┤
│ 我的匹配                 │  ← 核心功能直达
│ 我的求职看板             │
│ 移民档案                 │
│ 已保存筛选(Pro)        │
│ 升级 Pro                 │
├─────────────────────────┤
│ 设置(语言/通知)        │
│ 退出登录                 │
└─────────────────────────┘
```

- 未登录态下拉=登录/注册两枚(现状保留)。
- 不放:积分/消息中心/公开主页(论坛物,YAGNI)。
